Tracking a Credential Harvesting Campaign via Infrastructure Pivoting

Starting from a single phishing domain, this analysis traces adversary infrastructure across passive DNS, certificate transparency logs and WHOIS data — connecting three separate campaigns to the same threat actor pattern.

Starting Point

This analysis started with a single domain flagged by a colleague: a convincing Microsoft login clone that had been used in a targeted phishing email. The domain itself was unremarkable — registered ten days earlier, let's-encrypt certificate, hosted on a residential-looking IP in Germany.

What made it interesting was the registration pattern. The WHOIS data was obviously falsified, but in a consistent way — the same fake registrant name had been used on two other domains registered the same week. That's where the pivot started.

→
Methodology Note
All pivoting in this analysis used only publicly available sources: crt.sh, Shodan, VirusTotal graph, RiskIQ Community and URLScan. No proprietary threat intel feeds were used.

Infrastructure Pivoting

Step 1 — Certificate Transparency

Querying crt.sh for the initial domain revealed four additional certificates issued within the same 48-hour window, all sharing the same Let's Encrypt account fingerprint. Three of the resulting domains followed the same naming convention: [brand]-login-[random].com. 

# Query certificate transparency for domain cluster
curl "https://crt.sh/?q=%.microsofft-login.com&output=json" \
  | jq '.[] | {cn: .name_value, issued: .not_before}'OSINT

Step 2 — IP clustering via Shodan

All five domains resolved to IPs within the same /24 range — a hosting provider known for minimal abuse response. A Shodan search for that range with the filter http.favicon.hash:[value] returned 14 additional hosts running identical phishing kits, serving the same fake login page.

Diamond Model — Current Findings
Adversary
Unknown. Consistent OPSEC failures suggest low-to-medium sophistication. No prior attribution.
Capability
Off-the-shelf phishing kit, credential harvester with Telegram exfil. No custom malware observed.
Infrastructure
19 domains, single /24 hosting range, shared Let's Encrypt account, Telegram bot for exfil.
Victim
Targeted at corporate Microsoft 365 users. Lures reference German invoice and HR themes.

Confidence Assessment

Infrastructure links are strong — shared hosting, certificates and registration patterns tie the 19 domains together with high confidence. Attribution to a single actor is medium-high: the consistency of the kit, exfil method and targeting profile points to one operator, but the evidence does not rule out a shared phishing-as-a-service platform.

Conclusions

Infrastructure pivoting from a single domain uncovered a moderately active credential harvesting operation targeting German-speaking Microsoft 365 users. The actor shows consistent but unsophisticated OPSEC — predictable domain naming, shared hosting and reused kit versions all aid tracking.

The Telegram exfil channel and invoice-themed lures suggest financial motivation. No overlap with known tracked groups was identified; this appears to be an independent low-to-medium capability operator.
