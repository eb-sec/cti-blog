Building a Local CTI Pipeline: Why I Started and What I Am Building

A design and architecture post on MISP-CTI-Bridge — an open-source pipeline that connects process anomaly detection, external threat feeds and structured MISP event creation without cloud dependencies or commercial licensing.

The Problem I Kept Running Into

The more I read about threat intelligence, the more I noticed the same pattern: most of what gets written assumes you either work at a company with a six-figure threat intel budget, or you are content staying at the level of consuming other people's reports without producing any analysis of your own.

Neither suited me. I wanted to actually practice the discipline — collect indicators, correlate them, understand where they come from, map them to adversary behavior and store them in a way that builds up over time. Not just read about it.

The practical problem was fragmentation. MalwareBazaar has samples. VirusTotal has graph relationships. Shodan has infrastructure data. MITRE ATT&CK has the behavioral framework. MISP can tie it all together — but connecting these sources manually, every time, for every indicator, does not scale even in a lab environment. After the third time I copy-pasted a SHA256 hash across four browser tabs I decided to automate it.

Honest note
This project is currently in development. What follows is a description of the architecture and reasoning behind design decisions — not a finished product review. The code is on GitHub and will be updated as each module is completed.

What the Pipeline Does

MISP-CTI-Bridge is a local, open-source pipeline with five stages. Each stage is a separate Python module so any part can be replaced or extended independently. The entire stack runs on a single machine with Docker — no cloud accounts, no API keys that cost money, no vendor lock-in.

Pipeline Architecture
1. scanner.py — Process Anomaly Detection psutil · Windows
SHA256 hash of suspicious executable
2. misp_bridge.py — TI Lookup + MISP Event Creation PyMISP · MalwareBazaar · VT
Structured MISP event with ATT&CK tags
3. feed_sync.py — Feed Synchronisation MISP Feeds · Abuse.ch · CIRCL
Updated IOC knowledge base
4. reporter.py — HTML + Markdown Output Per-scan reports · Threat summaries

Design Decisions Worth Explaining

Why MISP and not a spreadsheet

The honest answer is: a spreadsheet would work for ten indicators. It stops working when you want to ask questions like "have I seen this hosting provider before" or "which of my stored indicators share a registrant with this new domain." MISP handles relationships between indicators natively. It also speaks STIX and TAXII, which means anything stored locally can eventually be shared with external platforms without reformatting.

The less obvious reason is that working with MISP forces you to think in structured intelligence terms. An event needs a threat level. Attributes need a type. Tags connect to frameworks. That friction is productive — it prevents the lazy habit of dumping raw IOCs somewhere and calling it intelligence.

Why local and not a managed service

Several managed MISP instances exist, including free community ones. The reason I run it locally comes down to two things. First, I wanted to understand how MISP actually works — running it in Docker and hitting problems with MySQL initialization timing taught me more about the platform than reading the documentation ever would. Second, a local instance means I control what goes in and what leaves. For a learning environment that matters less, but building the habit of thinking about data handling from the start seems worth it.

Why Python and not an existing automation tool

Tools like n8n or Shuffle could wire these services together without writing code. I chose Python for a simple reason: the pipeline is also a portfolio piece. Code that explains its own decisions through comments and structure communicates more about how you think than a no-code workflow screenshot.

On the code style
The project deliberately avoids over-engineering. No abstract base classes, no dependency injection frameworks, no enterprise patterns. Flat functions where they work, classes only where state is genuinely needed. Comments explain why decisions were made, not what the code does — the code does that itself.

The Intelligence Lifecycle in Practice

One of the things that attracted me to CTI as a discipline is that it has a real methodology — the intelligence lifecycle — that is easy to describe but difficult to actually follow through on. Most people stop at collection. The pipeline is partly an attempt to force myself through all six phases by making the later ones easier.

Phase What it means How the pipeline addresses it
Direction What are you trying to find out? Config file defines minimum severity thresholds and which processes are baseline-normal
Collection Gather raw data from sources scanner.py + MalwareBazaar + VirusTotal + MISP feeds
Processing Normalize and structure data misp_bridge.py normalizes findings into MISP events with typed attributes
Analysis Turn data into intelligence ATT&CK mapping, deduplication, feed correlation — this is the weakest phase currently
Dissemination Deliver findings to the right audience reporter.py generates HTML and Markdown with TLP labels
Feedback Refine requirements and iterate Not yet implemented — planned as a manual review step

The analysis phase is intentionally listed as the weakest. Automated ATT&CK tagging based on alert type is a heuristic, not analysis. Real analysis requires a human asking whether the observed behavior actually matches the technique, whether there are alternative explanations, and what confidence level the evidence supports. The pipeline gets you to the point where that question is easier to ask — it does not answer it for you.

What I Have Learned So Far

Building this has been a better education in CTI tooling than reading about it. A few things stand out.

MalwareBazaar is underused. The free API returns rich context — malware family, first seen date, tags, file type — with no authentication required. For a learning environment it is essentially a free tier of commercial threat intel.

MISP startup behavior is surprising. The Docker container initializes MySQL asynchronously, which means the application layer can be ready before the database is. The first time this caused a silent failure that took twenty minutes to diagnose. The fix — a retry loop with exponential backoff on the initial connection — is now in misp_bridge.py and is something I would not have thought to include if I had not hit the problem myself.

The VirusTotal rate limit is a design constraint, not an annoyance. Four requests per minute on a free key forces you to be selective about what you look up. That selectivity is actually good practice — in a real environment you want to query external services only for indicators that have already cleared an internal triage step.

⚠
Current Limitation
The scanner module is Windows-only. The path validation checks reference System32 and Windows Defender paths which do not translate to Linux. A cross-platform version is on the roadmap but is not the current priority.

What Comes Next

The immediate next step is finishing the feed_sync.py module and running the pipeline against a fresh MalwareBazaar sample to validate the full MISP event creation flow. That will produce the first real data to work with, and the first real post that is analysis rather than architecture description.

After that the most useful addition would be a Sigma rule generator — taking a MISP event and outputting a detection rule in Sigma format that can be used in any SIEM. That closes the loop from indicator discovery to deployed detection, which is ultimately what threat intelligence is for.

The code is on GitHub. If you are building something similar or have thoughts on the architecture, I am genuinely interested — the feedback loop is the phase I have the least infrastructure for right now.
